## Context

バックエンドは FastAPI + Redis を中心とした実装で、DDD/ヘキサゴナルの概念がコード構成に反映されている。新規メンバーが前提知識なしで参加できるよう、アプリ固有の読み方と一般的な概念の両方を解説する技術書レベルのドキュメントが必要である。今回の変更は仕様や API ではなく、理解促進のためのドキュメント追加が中心となる。

## Goals / Non-Goals

**Goals:**
- `docs/` を新設し、バックエンド実装の全体像・主要フロー・責務分離を丁寧に説明する
- FastAPI/Python/Redis/DDD/ヘキサゴナルの基礎を短く補足し、本アプリ内の対応箇所を明示する
- マーメイド記法による ER 図を含め、データモデルの理解を助ける

**Non-Goals:**
- アプリの実装や API の変更
- 既存仕様（OpenSpec）の改訂
- フロントエンドの詳細解説

## Decisions

- **ドキュメントの配置:** `docs/` 配下にまとめて配置し、README からリンクする前提で構成する。コード変更は不要で、既存のリポジトリ構成に影響しない。
  - *Alternative:* 既存の `openspec/` に追記する案 → 仕様と学習ガイドの目的が混在するため採用しない。
- **章立て:** 1冊の技術書として読めるよう、概要 → アーキテクチャ解説 → 実装の読み方 → 主要フロー → データモデル → 運用/デバッグの順に構成する。
  - *Alternative:* 断片的な FAQ 形式 → 学習導線が弱く全体像が掴みにくい。
- **図の表現:** Mermaid を採用し、ER 図・アーキテクチャ関係図をテキストで管理可能にする。
  - *Alternative:* 画像ファイル → バージョン管理や差分確認がしづらい。

## Risks / Trade-offs

- [分量が過多になり読むハードルが上がる] → 章ごとに「どの読者向けか」「読む順序」を明記する
- [実装とドキュメントの乖離] → 主要フローや図に対応するコードパスを明記し、更新しやすくする
