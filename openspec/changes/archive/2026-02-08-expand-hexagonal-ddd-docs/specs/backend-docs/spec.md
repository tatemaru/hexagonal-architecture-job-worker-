## MODIFIED Requirements

### Requirement: バックエンド技術書ドキュメントの提供

システムは `docs/` 配下に、バックエンド実装の理解を目的とした技術書レベルのドキュメントを提供しなければならない（SHALL）。ヘキサゴナルアーキテクチャと DDD に関する章は、基礎説明だけでなく背景・意図・パターン・図解を含む厚い内容でなければならない（SHALL）。

#### Scenario: ドキュメントの増補
- **WHEN** `docs/backend/02_ddd.md` と `docs/backend/03_hexagonal.md` を確認する
- **THEN** 概念の深掘り説明、図解、実装への対応付けが追加されている

### Requirement: ヘキサゴナルアーキテクチャの説明と対応付け

ドキュメントはヘキサゴナルアーキテクチャの概念（ポート/アダプター/ドメイン/ユースケース）を説明し、アプリ内の具体的なコード配置との対応関係を示さなければならない（SHALL）。Alistair Cockburn の Hexagonal Architecture をベースに、依存関係の方向、入力/出力アダプターの役割、変形パターン（例: 左右非対称の入口）を整理して説明しなければならない（SHALL）。

#### Scenario: Cockburn ベースの説明
- **WHEN** ヘキサゴナル章を読む
- **THEN** Cockburn の考え方を噛み砕いた説明と図解が含まれており、参考元として明記されている

### Requirement: DDD の主要概念の説明

ドキュメントは DDD の主要概念（エンティティ、値オブジェクト、集約、ドメインイベント）を簡潔に説明し、アプリ内の具体例を示さなければならない（SHALL）。さらに、概念間の関係（集約境界、イベントの意義、リポジトリとの分離）を図解付きで整理しなければならない（SHALL）。

#### Scenario: 概念関係の整理
- **WHEN** DDD 章を読む
- **THEN** 各概念の関係と責務が図解付きで説明されている
