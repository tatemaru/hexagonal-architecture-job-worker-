# 全体像とゴール

このバックエンドは「ジョブを作成し、非同期で実行し、状態を更新して通知する」というシンプルな問題を、学習用に DDD とヘキサゴナルアーキテクチャで整理した構成です。UI から見えるのは数本の API ですが、内部では以下の要素が明確に分離されています。

- **ドメイン**: ジョブの状態遷移や不変条件（何が許され、何が禁止か）
- **ユースケース**: ユーザーの意図（作成する/キャンセルする/一覧を見る）
- **ポート**: 依存性を逆転するための抽象インターフェース
- **アダプター**: HTTP/DB/Redis/通知など外部世界との接続
- **ワーカー**: Pub/Sub を通じてジョブを実行し、状態更新する独立プロセス

## このドキュメントで達成したいこと

- 「どこから読めばいいか」が分かる
- 「概念→実装」の対応が頭に入る
- 変更時にどの層を触るべきか判断できる

## システムのざっくり流れ

1. API でジョブ作成リクエスト（POST /api/jobs）
2. ユースケースが Job を作成し、イベントを Redis に配信
3. ワーカーがイベントを受け取りジョブを実行
4. 状態更新イベントが Redis に配信され、SSE でフロントに反映

次章から、必要最低限の技術要素とアーキテクチャ概念を押さえます。
